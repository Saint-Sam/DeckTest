# T3.1 Card Identity, Catalog, and DSL Specification

Status: frozen by Owner O4 approval on 2026-07-10, with the bounded PC-0005
corpus amendment incorporated

## Boundaries

`forge-carddef` owns stable catalog and validated mechanics data. `forge-cardc`
owns source parsing, validation, canonical emission, and deterministic database
building. `forge-cards` loads/query-validates the runtime database.
`forge-porttools` streams the pinned Scryfall metadata into a compact catalog.
Runtime game object ids remain in `forge-core`; printed mechanics never contain
live `ObjectId` or `PlayerId` values.

## Identity

- `PrintingId`: stable source printing id.
- `OracleId`: Scryfall `oracle_id`, or
  `source:<layout>:<printing-id>` when absent. Metrics report these fallback
  identities separately from source-supplied Oracle ids.
- Multi-face layouts keep ordered `CardFace` values under one `OracleId` when
  the source describes one game object.
- One `CardDefinition` supplies mechanics for an Oracle identity. Any number of
  `PrintingRecord`s may reference it.
- Every source record is cataloged; non-game/out-of-v1 records are explicitly
  classified and never count as playable.

## Source Language

```text
card "Llanowar Elves" {
  id: "<oracle-id>"
  layout: normal
  status: unverified_playable
  face "Llanowar Elves" {
    cost: "{G}"
    types: "Creature - Elf Druid"
    oracle: "{T}: Add {G}."
    power: "1"
    toughness: "1"
    keywords: []
    ability activated {
      costs: [tap_self()]
      effect: add_mana("{G}", you())
      mana_ability: true
    }
  }
}
```

Fields are order-independent but unique. Strings use JSON escapes. `//` starts
a comment only outside an atomic quoted string, so `Fire // Ice` is preserved.
Expressions are typed calls, lists, strings, signed integers, and booleans.
Bare symbols are accepted only in dedicated closed fields such as keyword
lists; they are rejected anywhere inside an ability expression. Unknown
operations never survive compilation.

The 100-card CP-DSL review corpus uses `unverified_playable`. It is a manually
reviewed language-stress corpus, not a claim that every recipe exactly
implements its retained Oracle text. `verified_playable` is reserved for a
definition whose behavior has passed card-specific semantic tests. T3.6 and
CP-PORT-20 own that promotion; round-trip, compilation, or generic engine
oracle success alone cannot promote a card.

## Validated IR

`CardDefinition` contains identity, layout, status, ordered faces, and
provenance. A face contains cost, type line, optional P/T/loyalty/defense,
oracle text, validated keywords, and abilities. Ability kinds are spell,
activated, triggered, static, and replacement. Ability fields are costs, event,
condition, timing, effect, and `mana_ability`.

The expression operation registry covers selectors/predicates, costs, events,
effects, duration, and state trackers. Each operation declares category,
arity, and a closed type for every argument position, including variadic
positions. Validation is recursive: it rejects an event in a cost field, a
selector where an amount is required, prose where an effect is required, and
any unknown operator or untyped argument.

PC-0005 adds closed presence-condition gates for activation timing, static
effects, replacements, and intervening-if triggers, plus typed value operations
for paid X, object/player counters, devotion, distinct-property counts, and
turn history. Dynamic mana uses an optional numeric `add_mana` argument instead
of embedding a symbolic amount in text. New operation variants are appended to
the registry so existing serialized operation discriminants and card databases
remain stable; the workspace compatibility suite guards this ordering.

The next bounded PC-0005 wave appends `unless_paid`, whose first two arguments
are the prevented effect and payer selector and whose remaining arguments are
closed costs, plus variadic `timing_all` for exact conjunctions of activation
restrictions. Source-zone activation timing uses a typed `zone_is` predicate
inside `timing_condition`; this context binds the predicate to the ability
source. Unsupported payer bindings, cost forms, and zones remain compile-time
or mapping failures rather than text-valued approximations.

The subsequent T3.3 breadth wave appends three structural families without
promoting runtime or card semantics. `add_restricted_mana` retains a
closed-vocabulary legacy spend restriction alongside the mana pattern,
recipient, and typed amount. `library_dig` retains the affected library owner, typed dig
and selection counts, closed card filter, both destinations, and a canonical
validated option record for visibility, ordering, remembering, tapping, and
face-down handling. `event_chapter` and `chapter` retain the chapter number,
final chapter, and resolved DB effect so multi-counter chapter crossings are
not approximated as an equality check on the current lore-counter total.
Source-bound `move_zone_from` effects retain both the required origin and
destination rather than approximating a closed-origin move as unconditional.
Creature-type replacement uses the closed `creature_subtypes` value of
`remove_type` before appending replacement types. Unsupported option shapes,
unresolved links, and open restriction values remain mapper failures.

The next T3.3 wave appends operations after the pinned serialized registry
prefix. `event_cast_targeting` retains the spell selector, any-target selector,
activating-player selector, and cast-versus-copy mode. `regenerate_shield`
retains regeneration as a distinct effect instead of approximating prevention.
`hidden_information` preserves legacy hidden-selection metadata around the
typed zone effect, including public-zone replacement cases. Generic
`Optional=True` activated and DB effects lower to a bounded `choose_up_to`
around the primary effect while subsequent cleanup/subability chains remain
outside that choice; Dig keeps its distinct optional-selection semantics.

Layer integration scenarios live in a separate ten-definition source packet.
`forge-cardc` builds that packet into a versioned database, `forge-cards` loads
it, and the arena lowers only its validated `layer_effect` trees into core
continuous effects. Fixture setup supplies object/player bindings and expected
outcomes, but contains no independently executable effect seeds.

## Mana And Types

Mana supports generic, W/U/B/R/G/C, X/Y/Z, snow, hybrid, monocolored hybrid,
Phyrexian, hybrid-Phyrexian, and half symbols. Card/supertypes are closed enums;
subtypes are validated non-empty source data. Keyword ids are checked against a
compiler registry generated from the supported keyword inventory.

## Database

`carddb.bin` is `FORGECDB` magic + little-endian schema version + bincode-v2
serde payload. The payload contains source provenance, sorted identity catalog,
sorted printings, and sorted mechanics definitions. Absolute source paths and
timestamps are excluded. `carddb.index.json` contains inspectable identity/name
and printing lookup rows. `forge-cards` rejects bad magic, unsupported versions,
trailing data, duplicate ids, dangling printing references, and unsorted input.

## Catalog Import

The importer streams the top-level Scryfall array and retains English metadata
only. It records source path, Scryfall update timestamp, local SHA-256, and
generator version. It imports no image URLs or image payloads into runtime
assets. Catalog metrics are generated from the imported records, never typed.

## CP-DSL Strata

The 100-card packet covers at least these 25 strata: vanilla/keywords,
instant/sorcery, modal choices, activated/mana/loyalty, triggered/delayed,
replacement/prevention, continuous/layers, targeting/restrictions, zones/LKI,
calculated values, alternate/additional costs, counters/tokens/copy,
attachments, multiplayer/Commander/voting, split/fuse, transform/day-night,
modal DFC, adventure, meld, flip, saga/class/case/leveler, unusual mana,
cast-from-zone mechanics, dungeon/initiative/monarch/energy, and turn/combat
mechanics. These names are a closed required set, with exactly four reviewed
definitions in each stratum. Catalog-only records cannot be mechanics
definitions; their classification and exclusion from playable coverage are
verified separately over the complete imported catalog.

The packet freezes identity, grammar, closed typing, canonical emission, and
database contracts. It does not freeze the semantic correctness of the 100
stress recipes. Obvious language expressiveness gaps discovered during review
must still be corrected before O4; card-specific fidelity remains fail-closed
through `unverified_playable` until later semantic evidence exists.

## Fail-closed Verification

CP-DSL requires 100/100 round-trips and at least 50 explicitly tagged recursive
argument diagnostic cases. That subset covers every closed `ArgumentKind`,
multiple nesting depths, variadic positions, bare symbols, prose misuse, and
category-correct operations in the wrong argument position. The gate also
requires three identical clean-build database hashes, at least 90% curated
mutant kill rate with no P0/P1 survivor, 100% English printing import, 100%
identity classification, zero dangling catalog references, and at least 2,400
verified address-sanitizer worker-seconds across all five local fuzz targets.

Mutation evidence includes a passing unmutated control, each expected killing
test, full command output, and log hashes. Fuzz evidence includes actual elapsed
fuzzer time and final libFuzzer statistics from retained, hashed worker logs.
The final packet is produced from a clean detached commit with isolated target
directories and no GitHub Actions. Cross-target evidence must contain linked
artifacts, not metadata-only checks: a WASM module and Android, iOS, and Windows
static libraries, each with a retained build log, nonzero size, expected magic,
and SHA-256. O4 Owner approval is required before T3.2 mass translation begins.
