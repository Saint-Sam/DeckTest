# T2.2 Triggered Abilities

## Goal

T2.2 adds a declarative CR 603 triggered-ability substrate on top of the T2.1
event stream. Trigger definitions are inert data compiled from card IR in later
tiers; the kernel interprets predicates over `GameEvent` and never calls
card-specific Rust code.

## Public API

- `TriggerId`: stable handle for one registered trigger definition.
- `GameEventKind`: subscription key derived from each `GameEvent`.
- `TriggerDefinition`: controller, optional source, condition, intervening-if,
  and duration.
- `TriggerCondition`: declarative event predicates for object moves, steps, life
  changes, damage, stack resolution, and coarse event kinds.
- `TriggerInterveningIf`: declarative state predicates checked when a trigger
  would queue.
- `TriggerDuration`: persistent or delayed-once subscriptions.
- `PendingTriggeredAbility`: queued trigger instance waiting for the next
  priority window.
- `Action::RegisterTriggeredAbility`: installs one data-only subscription.
- `Action::PutPendingTriggeredAbilitiesOnStack`: drains pending triggers in
  APNAP order as `StackObjectKind::TriggeredAbility` stack entries.

## Rules

Successful events scan matching subscriptions by `GameEventKind`. A matching
condition and true intervening-if predicate enqueue one pending trigger instance.
Delayed-once definitions are removed after their first queued instance.

Pending triggers block normal priority actions. `legal_actions` returns
`PutPendingTriggeredAbilitiesOnStack` before `PassPriority`, and direct priority
passes fail with `StateError::PendingTriggeredAbilities` until the queue is
drained. Draining preserves APNAP order: active-player triggers are placed lower
on the stack, then nonactive players in turn order, preserving source order
within each controller.

Trigger lifecycle events are emitted for registration, queueing, and stack
placement. These events do not recursively trigger themselves.

## Determinism

Trigger subscriptions, pending trigger instances, deferred priority, trigger IDs
on stack entries and resolution records, and trigger lifecycle events are covered
by canonical bytes and streaming deterministic hashes.

## Out Of Scope

T2.2 does not compile real card files into trigger definitions, resolve trigger
effects, implement optional "may" choices, or implement target selection for
triggered abilities. Those are later card IR, effect-resolution, and targeting
tasks. T2.3 handles replacement/prevention effects.

## Tests

Unit coverage proves object-move queueing, priority blocking, stack placement,
APNAP ordering, intervening-if gating, delayed-once retirement, and canonical
hash consistency with trigger state.
