# T2.1 Event System Substrate

## Goal

T2.1 adds a typed, data-only event stream to `forge-core` so later triggered
abilities, replacement/prevention effects, narration, and replay diagnostics can
observe kernel mutations without closures or card-specific engine code.

## Public API

- `EVENT_RING_CAPACITY`: maximum current-turn events retained by `GameState`.
- `GameEvent`: typed payload enum for successful kernel mutations.
- `EventRecord`: `(sequence, turn, event)` record stored in deterministic order.
- `EventCursor`: replay position containing the current turn and next sequence.
- `EventReplayError`: stale, future, or wrong-turn cursor errors.
- `GameState::events_this_turn()`: current retained event slice.
- `GameState::event_cursor()`: cursor after the latest retained event.
- `GameState::events_since(cursor)`: current-turn replay from a cursor.

## Emission Rules

Events are emitted only after a mutation validates and succeeds. Failed actions
must not append events, advance event cursors, or change deterministic hashes.

Events are emitted from private mutation helpers rather than by diffing full
state or by relying only on `apply`, so internal rules work in later tiers is
observable through the same substrate.

Priority-pass churn is intentionally not part of the event stream. The existing
action replay records those passes, while T2.2/T2.3 trigger and replacement
matching needs semantic game events such as moves, damage, life changes, stack
resolution, step boundaries, and state-based actions.

The event stream is bounded to the current turn. Starting a new turn clears the
retained event slice and emits `TurnStarted` followed by the new step events,
while the monotonic sequence counter continues increasing for replay ordering.

## Determinism

Event records are part of canonical bytes and streaming deterministic hashes.
Two replays that reach the same board but emit different event histories should
not be considered identical by low-level diagnostics.

## Out Of Scope

T2.1 does not implement trigger subscription tables, replacement/prevention
interception, delayed triggers, card IR compilation, or oracle scenario grammar
for event expectations. Those begin in T2.2 and T2.3.

## Tests

Unit coverage must prove that setup/object movement emits ordered events,
failed actions leave the event stream unchanged, the ring capacity is enforced,
stale cursors are rejected, new turns reset retained events, and event state is
covered by the existing canonical hash consistency tests.
