# T3.5 Capability-Specific Runtime Smoke

Status: first bounded slice implemented and verified locally

## Contract

`forge-testkit runtime-smoke <FILE-OR-DIRECTORY>` parses compiler-valid `.frs`
definitions and emits one of three typed results: `passed`,
`unsupported_setup`, or `failed`. Unsupported setup is a distinct result with a
stable reason code, is excluded from the pass count, and makes the command exit
nonzero. Compiler and file errors are failures, not unsupported results.

The first supported family is intentionally narrow:

- normal, single-face cards whose only card type is `Instant` or `Sorcery`;
- no keywords and exactly one unconditional spell ability;
- printed mana containing only generic and ordinary colored symbols;
- literal `gain_life` and `lose_life` effects, optionally under `sequence`;
- player binding through only `you()` or `opponent()`.

The synthesizer creates two players, seeds and keeps real seven-card opening
hands, preserves a draw-step card for sorceries, enters a legal cast window,
pays the printed cost, casts through `Action::CastSpell`, and resolves through
production priority passes. It asserts the spell's stack transition and owner
graveyard destination. Supported life effects execute immediately after generic
spell resolution through `Action::GainLife` and `Action::LoseLife`, with no
intervening gameplay action. This ordering is bounded to an otherwise empty
setup with no observers; it is not a general card-resolution hook.

After every successful production action, the harness checks zone conservation,
allocated versus streaming deterministic hashes, player-view construction, and
life/poison sanity. It then asserts exact final life totals, the final spell
zone, and an in-progress game outcome.

Stable unsupported codes cover classification, layout, face count, card type,
keyword semantics, ability shape, mana symbols, effect operations, effect
arguments, effect amounts, player selectors, and bounded setup limits. Runtime
failures separately code production-action rejection, unexpected outcomes,
invariant violations, destination mismatches, and life-total mismatches.

## Local Evidence

All commands ran locally with Cargo `--offline`; no network, install, remote CI,
commit, or push was used.

- `cargo test --offline -p forge-testkit runtime_smoke -- --nocapture`: 6 passed,
  0 failed across the library and CLI. The supported fixture executed
  `gain_life,lose_life` in 37 production actions, reached `owner_graveyard`, and
  produced hash `8990916786298655732`. The draw fixture emitted
  `unsupported_effect_operation` and did not pass.
- `cargo test --offline -p forge-testkit`: 25 passed, 0 failed, plus 0 doc tests.
- `cargo clippy --offline -p forge-testkit --all-targets -- -D warnings`: passed.
- `cargo fmt -p forge-testkit -- --check` and focused `git diff --check`: passed.
- `cargo run --offline -q -p forge-testkit -- runtime-smoke
  target/translated-cards/n/nourish.frs`: 1 passed, 0 unsupported, 0 failed;
  31 production actions, `gain_life`, owner graveyard, hash
  `17086081742352476644`.
- `cargo run --offline -q -p forge-testkit -- runtime-smoke
  target/translated-cards/h/heroes_reunion.frs`: expected nonzero result;
  exit 2, with 0 passed, 1 unsupported, 0 failed and
  `unsupported_player_selector`.

This evidence proves only the listed capability family. It does not claim that
all translated cards, targeted life effects, dynamic amounts, other spell
effects, permanents, triggered/activated/static abilities, alternate costs, or
keyword semantics pass runtime smoke.
