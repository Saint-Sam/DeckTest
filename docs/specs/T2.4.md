# T2.4 Continuous Effects Layer System

T2.4 adds the first CR 613 continuous-effects engine. Effects are declarative
data installed on `GameState`; the kernel recomputes effective object
characteristics on demand from stored base values plus registered effects.

## Public Surface

- `ContinuousEffectId`: stable handle for one registered continuous effect.
- `ObjectColors` and `ObjectTypes`: compact represented color/type sets.
- `ObjectCharacteristics`: effective controller, colors, types, text marker,
  and creature characteristics after layers.
- `ContinuousEffectTarget`: one object or all objects.
- `ContinuousEffectOperation`: layer-specific operations for copy, control,
  text marker, type, color, ability, and power/toughness sublayers.
- `ContinuousEffectDefinition`: data-only controller/source/target/operation,
  duration, timestamp, and dependency edges.
- `Action::RegisterContinuousEffect`: validates and installs one definition.
- `GameState::object_characteristics`, `object_controller`, and
  `creature_characteristics`: recompute effective values through layers.

## Layer Order

Queries apply effects in CR 613 order: copy, control, text, type, color,
ability, then power/toughness 7a, 7b, 7c, and 7d. Within one layer or sublayer,
effects use timestamp order with deterministic effect ID as the tie-breaker.
Explicit dependency edges reorder only effects that are applicable to the same
object in the same layer. Cross-layer dependencies are ignored by construction
because the fixed layer order still dominates.

The layer-1 copy operation copies stored base creature values only. It does not
copy later source modifiers.

## Rules Integration

Rules code that needs a controller or creature characteristics now goes through
effective queries. Casting ownership checks, attack/block legality, intervening
trigger controller checks, lethal damage, zero-toughness SBAs, and combat keyword
queries therefore see layer results instead of stored printed values.

Stored object records still hold base creature characteristics. Canonical bytes
and streaming hashes serialize the stored base values plus the registered
continuous-effect list, so effective queries remain deterministic without
persisting derived characteristics.

## Testkit Surface

RON scenarios support:

- `register_continuous_effect`
- `assert_characteristics`
- `expect.characteristics`

Expectations can check controller, colors, types, creature/noncreature status,
power/toughness, keywords, and text marker through the public query path.

## Coverage

Unit tests under `tests::layers` cover dependency ordering, power/toughness
sublayer order, copy values, effective control, and hash participation.

Oracle scenarios `tests/oracle/layers/layers_001` through `layers_080` cover
timestamps, same-layer dependencies, cross-sublayer dependency isolation,
all-object effects, copy effects, controller changes, type/color/text/ability
layers, power/toughness 7a-7d, SBA interactions, and mid-script invalidation.

## Known Boundaries

This task implements a deterministic data substrate, not a card compiler. The
`forge-carddef` layer remains bootstrap-only, and real card text lowering is a
later T3 responsibility. No memoization cache is present yet, which makes the
T2.4 invalidation story simple: every query recomputes from current state.

CP-LAYERS remains mandatory before T2.5+ depends on this design.
